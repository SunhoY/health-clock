# UX Guide (Mobile-First)

## 목적
이 문서는 모바일 앱/모바일 웹앱 환경에서 사용자가 빠르게 이해하고 행동할 수 있는 UX 기준을 정의한다.

## 핵심 원칙
1. 장문 설명보다 직관적 UI를 우선한다.
2. 화면에서 "읽게" 만들기보다 "보자마자 알게" 만든다.
3. 한 화면에는 한 가지 핵심 행동만 강조한다.
4. 텍스트는 최소화하고, 구조/아이콘/위계/상태로 의미를 전달한다.

## 모바일 앱 고려사항 (명시)
- 모바일 환경에서는 긴 설명 텍스트(Verbose Text) 의존을 지양한다.
- 필수 안내가 아니라면 문단형 설명 대신 다음을 우선한다.
  - 명확한 제목
  - 짧은 보조 문구(1~2줄)
  - 직관적 아이콘/레이블
  - 즉시 이해 가능한 버튼 문구
- 사용자가 읽어야만 이해되는 화면은 UX 결함으로 간주한다.

## 모바일 웹앱 UX 가이드

### 1) 정보 구조
- 첫 화면에서 사용자 목적(기록, 확인, 수정)을 즉시 선택할 수 있어야 한다.
- 깊은 메뉴 구조를 피하고, 최대 2~3단계 내에서 주요 기능에 도달하게 한다.
- 우선순위가 낮은 기능은 "더보기" 영역으로 분리한다.

### 2) 텍스트/카피
- 제목은 짧고 동사 중심으로 작성한다. 예: "오늘 기록하기", "수면 시간 수정".
- 본문 안내는 한 문장 20자 내외, 최대 2줄을 권장한다.
- 버튼은 행동이 드러나는 문구를 사용한다. 예: "저장", "다음", "완료".
- 에러 문구는 원인 + 해결 방법을 짧게 제공한다.

### 3) 터치 인터랙션
- 터치 영역은 최소 44x44px 이상으로 구성한다.
- 주요 CTA는 엄지 도달 영역(하단)에 배치한다.
- 스와이프/롱프레스 같은 숨은 제스처에 핵심 기능을 의존하지 않는다.

### 3-1) 프로젝트 기본 규칙 (Health Clock)
- 랜딩 화면의 기본 구성은 `앱 제목 + 단일 CTA`를 우선한다.
- 광고성/과도한 유도성 카피는 기본값에서 제외한다.
- 모바일에서는 주요 CTA를 하단 엄지 영역에 고정 배치하고, safe area를 고려한다.
- 데스크톱/태블릿에서는 동일 CTA를 문맥상 자연스러운 위치(중앙/본문 하단)로 배치한다.

### 4) 시각적 위계
- 타이포 크기/굵기/여백으로 우선순위를 명확히 만든다.
- 한 화면의 핵심 색상 포인트는 1~2개로 제한한다.
- 상태(성공/오류/주의/비활성)를 색상 + 아이콘 + 텍스트로 중복 전달한다.

### 5) 폼 입력
- 입력 항목 수를 최소화하고, 가능한 자동완성/기본값을 제공한다.
- 모바일 키보드 타입(number, email 등)을 필드 목적에 맞춰 지정한다.
- 실시간 유효성 검사를 제공하되 입력 흐름을 방해하지 않는다.

### 6) 성능 체감
- 초기 로딩은 핵심 콘텐츠 우선 렌더링으로 체감 속도를 확보한다.
- 로딩 상태는 skeleton/spinner 등으로 즉시 피드백한다.
- 이미지/아이콘은 모바일 해상도에 맞게 최적화한다.

### 7) 접근성
- 텍스트 대비(contrast)를 확보하고 작은 글씨 남용을 피한다.
- 스크린리더를 위한 의미 있는 레이블과 순서를 유지한다.
- 상태 변화는 시각 요소만이 아니라 텍스트로도 전달한다.

### 8) 반응형/디바이스 대응
- 최소 지원 뷰포트를 기준으로 레이아웃이 깨지지 않아야 한다.
- 세로 모드 중심으로 설계하되 가로 모드에서 핵심 기능 접근은 보장한다.
- 안전 영역(safe area, 노치/하단 홈 인디케이터)을 고려한다.

### 9) PWA/모바일 웹앱 특화
- 홈 화면 추가(Install) 동선을 방해하지 않게 안내한다.
- 네트워크 불안정 상황을 고려해 재시도/오프라인 안내를 제공한다.
- 앱처럼 보이되, 웹 특성(브라우저 뒤로가기/새로고침)과 충돌하지 않게 설계한다.

## 디자인 리뷰 체크리스트
- 이 화면은 긴 설명 없이도 사용 목적이 3초 내 이해되는가?
- 핵심 행동 버튼이 첫 시야와 엄지 영역에 있는가?
- 한 화면에 경쟁하는 주요 CTA가 2개를 넘지 않는가?
- 텍스트를 줄여도 구조적으로 의미가 유지되는가?
- 모바일 네트워크/저성능 기기에서도 사용 흐름이 끊기지 않는가?
